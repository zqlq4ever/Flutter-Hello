# 性能/体积基线采集（Android）

本文件用于在开始优化前建立**可复现、可对比**的基线数据。建议每次做完一个阶段（例如“启动闪屏修复”“列表重构”“R8/资源压缩”）后都复采一次并对比差异。

## 0. 约定
- **构建模式**：性能数据以 `profile` 为主；体积以 `release` 为主。
- **设备**：尽量固定同一台真机（同系统版本、同电量/温度、同网络环境）。
- **记录方式**：把每次采集结果追加到 `tooling/perf_baseline_log.md`（你可以自行创建），避免散落到聊天记录里。

## 1. 启动基线（冷启动/首帧）
### 1.1 目标
- **消除黑/白闪屏**：启动过程中不出现明显背景突变。
- **缩短首帧时间**：以 Flutter `TimeToFirstFrame` 为主观与客观对照指标。

### 1.2 采集步骤
1. 以 profile 运行：

```bash
flutter run --profile
```

2. 打开 DevTools（IDE 或命令行提示），在 **Performance** 面板观察：
   - `Time to first frame`
   - `Raster/Build` 时长是否出现明显尖峰

3. **冷启动闪屏复现**：完全杀进程后启动 3 次，记录是否出现黑/白闪一下（建议录屏）。

- 设备：____
- Flutter/Dart：____
- 分支/提交：____
- `TimeToFirstFrame`（3 次）：____ / ____ / ____
- 是否闪屏：是/否（附录屏文件名）

## 2. 滚动/列表流畅度基线（掉帧/jank）
### 2.1 目标
- 以“长列表页”为样本，把掉帧（jank）降低到可接受水平。
- 优先看“**连续快速滚动**”时的 frame chart（最容易暴露布局/重建问题）。

### 2.2 采集步骤
1. profile 运行后进入样本页面（建议先用：通讯录页、通话记录页等长列表）。
2. DevTools → Performance：
   - 开启 frame chart
   - 快速滚动 10–15 秒
   - 记录 jank 帧数量、最长帧耗时区间（Build/Raster）
3. 如果重构目标是“去嵌套滚动/去 shrinkWrap”，重点对比重构前后：
   - jank 帧数量
   - 99th percentile frame time（如果面板可见）

### 2.3 记录模板
- 样本页面：____
- 操作描述：快速滚动 __ 秒
- jank 帧数量：____
- 主要瓶颈：Build / Raster（圈出）

## 3. 包体积基线（APK/AAB）
### 3.1 目标
- 建立当前基线包体积，后续针对依赖裁剪、R8、资源压缩做对比。

### 3.2 采集步骤
1. 构建 release（先选一种即可，建议 AAB）：

```bash
flutter build appbundle --release
```

2. 记录输出文件大小：
   - `build/app/outputs/bundle/release/app-release.aab`

3. 可选：输出依赖/资源拆分分析（如果你后续要做更细的体积治理）。
